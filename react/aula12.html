Flux - Arquitetura e Padrão de Projeto

História: O facebook tinha problemas com a falta de sincronismo no sininho das notificações, o que gerava uma série de inconsistências. Até então havia alguns patterns e modelos (MVC) para trafegar dados em comunicações web de maneira multidirecional.
MODEL <<==>> VIEW

Flux é uma arquitetura criada pelo facebook para a comunicação entre componentes.
É um padrão de projeto para tráfego de dados de maneira unidirecional.
1.ACTION <<==>> 2.DISPATCHER <<==>> 3.STORE <<==>> 4.VIEW
Segue o fluxo 1,2,3,4...volta para 1 (da view volta para action).

Action

É como um telegráfo, ele formata a mensagem a ser enviada.
Por exemplo, quando é necessário alterar o estado de algo, é ela que formata a mensagem.
Ele também envia a mensagem para o dispatcher.

Dispatcher

É como um telefonista, ele sabe todos os callbacks para diferentes stores.

Store

É como um gerente super controlador. Ele guarda a informação (Estados) e todas as alterações têm que ser feitas por ele, mais ninguém.

View

É como um gerente intermediário (middleware) que recebe as notificações da store e passa os dados para as visões abaixo dela.
Não confundir a view com a tela em si.
A view é como se fosse um middleware entre o store e a tela.
É ele que vê as alterações de estado e pede ao React para fazer a alteração no virtual DOM. É ele que faz essas notificações.

Resumindo: Não é possível alterar diretamente o estado, é necessário passar uma mensagem pedindo alteração. Essa é a diferença do fluxo unidirecional.

Bibliotecas baseadas em Flux
Facebook disponibiliza bibliotecas no NPM para construção de implementações de Flux.
Servem para comunicações entre componentes e centralização da informação.
"Flux libraries are like glasses: you'll know when you need them." - Dan Abramov
  • Redux (Mais popular)
  • Reflux (Mais simples que o Redux)
  • Mobx
  • Vuex (Usado no Vue, Baseada em Redux e Elm)
  • NgRx/Store (Mais usada no Angular, Baseada em Redux e RxJS)
